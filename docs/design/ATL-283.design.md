## 1. Architecture Overview

### 파일 구조

```
admin-ui/
├── components/
│   └── Sidebar.tsx                          ← [수정] A/B Tests 링크 추가
└── app/(admin)/
    ├── ab-tests/
    │   └── page.tsx                         ← [신규] 메인 구현
    └── __tests__/
        └── ABTestsPage.test.tsx             ← [신규] 컴포넌트 테스트
```

### 컴포넌트 계층

```
ABTestsPage (default export)
├── StatusBadge({ status })                  — 상태 배지 (draft/running/paused/completed/stopped)
├── CreateDialog({ onClose, onCreated })     — 실험 생성 모달
├── ResultsPanel({ test, onClose })          — 결과 인라인 패널 (행 확장)
└── PromoteDialog({ test, onClose })         — 승자 선택 확인 모달
```

모든 서브컴포넌트는 `page.tsx` 단일 파일에 named function으로 정의한다 (PromptsPage, KeysPage 패턴과 동일).

### API 클라이언트 사용 (`lib/api.ts` — 수정 불필요)

| 동작 | API 메서드 | 반환 타입 |
|---|---|---|
| 목록 조회 | `abTests.list()` | `ABTest[]` |
| 단건 조회 | `abTests.get(id)` | `ABTest` |
| 실험 생성 | `abTests.create(payload)` | `ABTest` |
| 결과 조회 | `abTests.results(id)` | `Record<string, unknown>` (로컬 타입 캐스팅 필요) |
| 시작 | `abTests.start(id)` | `{ status: string }` |
| 일시정지 | `abTests.pause(id)` | `{ status: string }` |
| 중지 | `abTests.stop(id)` | `{ status: string }` |
| 승자 지정 | `abTests.promote(id, winner)` | `{ status: string; winner: string }` |

## 2. TypeScript Types

### 2.1 `lib/api.ts` 기존 타입 (그대로 사용)

```typescript
// lib/api.ts에 이미 정의되어 있음 — 수정 불필요
export interface ABTestTrafficSplit {
  variant: string;
  weight: number;
  // 주의: lib/api.ts의 ABTestTrafficSplit에는 model 필드가 없다.
  // 실제 백엔드 TrafficSplit에는 model 필드가 있으므로 런타임 캐스팅 필요.
}

export interface ABTestTarget {
  model: string;       // 주의: 백엔드 Target 구조에는 model이 없다.
  sample_rate?: number;
}

export interface ABTest {
  id: string;
  name: string;
  status: string;      // "draft" | "running" | "paused" | "completed" | "stopped"
  traffic_split: ABTestTrafficSplit[];
  target: ABTestTarget;
  success_metrics?: string[];
  min_samples?: number;
  confidence_level?: number;
  start_at?: string;
  end_at?: string;
  // 주의: 백엔드 Experiment에는 winner, created_at, updated_at이 있으나 lib/api.ts에 미선언.
  // 페이지 내 로컬 타입 ExtendedABTest로 보강.
}

export interface CreateABTestPayload {
  name: string;
  traffic_split: ABTestTrafficSplit[];
  target: ABTestTarget;
  success_metrics?: string[];
  min_samples?: number;
  confidence_level?: number;
  start_at?: string;
  end_at?: string;
}
```

### 2.2 페이지 내 로컬 타입 (`app/(admin)/ab-tests/page.tsx`)

`lib/api.ts`의 ABTest 타입이 백엔드 실제 응답 필드를 일부 누락하므로 페이지 내에서 보강 타입을 정의한다.

```typescript
// lib/api.ts의 ABTest에 없는 백엔드 필드를 포함한 확장 타입
interface TrafficSplitWithModel {
  variant: string;
  model: string;       // 백엔드 TrafficSplit.Model
  weight: number;
}

// abTests.results() 응답을 Record<string,unknown>에서 캐스팅하기 위한 로컬 타입
interface VariantResult {
  model: string;
  samples: number;
  latency_p95_ms: number;
  avg_cost_per_request: number;
  error_rate: number;
}

interface MetricSignificance {
  p_value: number;
  significant: boolean;
  improvement_pct: number;
}

interface AnalysisResult {
  test_id: string;
  status: string;
  winner?: string;
  results: Record<string, VariantResult>;
  statistical_significance?: Record<string, MetricSignificance>;
  recommendation?: string;
}

// CreateDialog 폼 내부 상태
interface TrafficSplitFormRow {
  variant: string;
  model: string;
  weight: number;
}

interface CreateFormState {
  name: string;
  splits: TrafficSplitFormRow[];   // 최소 2개
  sample_rate: number;             // 0.0–1.0, default 1.0
  min_samples: number;             // default 1000
  confidence_level: number;        // 0.0–1.0, default 0.95
  start_at: string;                // datetime-local input value, ""이면 미설정
  end_at: string;
}
```

**타입 불일치 처리 전략**:
- `ABTest.traffic_split` → `TrafficSplitWithModel[]`로 캐스팅해 `model` 필드에 접근
- `abTests.results()` 반환값 → `as unknown as AnalysisResult`로 캐스팅
- `created_at`, `updated_at`, `winner`는 `(test as unknown as { created_at: string }).created_at` 패턴으로 선택적 접근하거나 `ExtendedABTest` 인터페이스를 통해 통일

**상태**:
```typescript
const [showCreate, setShowCreate] = useState(false);
const [resultsTest, setResultsTest] = useState<ABTest | null>(null);
const [promoteTest, setPromoteTest] = useState<ABTest | null>(null);
```

**데이터 페칭**:
```typescript
const qc = useQueryClient();

const { data: testList = [], isLoading } = useQuery({
  queryKey: ["ab-tests"],
  queryFn: abTests.list,
  refetchInterval: 30_000,
});
```

**상태 전환 Mutations**:
```typescript
const startMutation = useMutation({
  mutationFn: (id: string) => abTests.start(id),
  onSuccess: () => qc.invalidateQueries({ queryKey: ["ab-tests"] }),
  onError: (e: Error) => setTransitionError(e.message),
});

const pauseMutation = useMutation({ /* 동일 패턴 */ });
const stopMutation  = useMutation({ /* 동일 패턴 */ });
```

트랜지션 에러는 `const [transitionError, setTransitionError] = useState<string | null>(null)`에 저장하고 목록 상단에 표시한다.

**상태별 액션 버튼 렌더링 로직**:

```typescript
function ActionButtons({ test }: { test: ABTest }) {
  return (
    <div className="flex gap-1.5 flex-wrap">
      {(test.status === "draft" || test.status === "paused") && (
        <button onClick={() => startMutation.mutate(test.id)} ...>Start</button>
      )}
      {test.status === "running" && (
        <button onClick={() => pauseMutation.mutate(test.id)} ...>Pause</button>
      )}
      {(test.status === "running" || test.status === "paused") && (
        <button onClick={() => stopMutation.mutate(test.id)} ...>Stop</button>
      )}
      {(test.status === "running" || test.status === "completed") && (
        <button onClick={() => setPromoteTest(test)} ...>Promote</button>
      )}
      {test.status !== "draft" && (
        <button onClick={() => handleResultsToggle(test)} ...>
          {resultsTest?.id === test.id ? "Hide Results" : "Results"}
        </button>
      )}
    </div>
  );
}
```

`handleResultsToggle(test)`: `resultsTest?.id === test.id`이면 `null`로 설정(토글), 아니면 해당 테스트로 설정.

**테이블 구조**:

```
| Name | Status | Traffic Split | Created At | Actions |
```

- Traffic Split 열: `splits.map(s => `${s.variant}:${s.weight}%`).join(" / ")` 형태로 요약 표시
- Actions 열: `ActionButtons` 컴포넌트 렌더링

**인라인 Results 패널** (PromptsPage의 VersionPanel 패턴과 동일):
```tsx
<React.Fragment key={test.id}>

</React.Fragment>
```

**Empty State**:
```tsx
<div className="bg-white rounded-xl border border-slate-200 p-8 text-center space-y-3">
```

### 3.3 CreateDialog

**역할**: 새 실험 생성 모달. weight 합산 100 검증 포함.

**Props**:
```typescript
function CreateDialog({ onClose, onCreated }: {
  onClose: () => void;
  onCreated: () => void;
})
```

**내부 상태**:
```typescript
const [form, setForm] = useState<CreateFormState>({
  name: "",
  splits: [
    { variant: "control",   model: "", weight: 50 },
    { variant: "treatment", model: "", weight: 50 },
  ],
  sample_rate: 1.0,
  min_samples: 1000,
  confidence_level: 0.95,
  start_at: "",
  end_at: "",
});
const [error, setError] = useState("");
```

**weight 합산 검증**:
```typescript
const totalWeight = form.splits.reduce((sum, s) => sum + s.weight, 0);
const weightValid = totalWeight === 100;
const canSubmit = form.name.trim() !== ""
               && form.splits.length >= 2
               && form.splits.every(s => s.variant.trim() !== "" && s.model.trim() !== "")
               && weightValid
               && !mutation.isPending;
```

Submit 버튼: `disabled={!canSubmit}`.

`!weightValid`일 때 폼 하단에 경고 문구 표시:
```tsx
{!weightValid && (
  <p className="text-xs text-red-500">
    Traffic weights must sum to 100 (current: {totalWeight})
  </p>
)}
```

**동적 splits 배열 관리**:
```typescript
// 행 추가 (최대 제한 없음)
function addSplit() {
  setForm(f => ({
    ...f,
    splits: [...f.splits, { variant: "", model: "", weight: 0 }]
  }));
}

// 행 삭제 (최소 2개 유지)
function removeSplit(idx: number) {
  setForm(f => ({
    ...f,
    splits: f.splits.filter((_, i) => i !== idx)
  }));
}

// 행 필드 업데이트
function updateSplit(idx: number, field: keyof TrafficSplitFormRow, value: string | number) {
  setForm(f => ({
    ...f,
    splits: f.splits.map((s, i) => i === idx ? { ...s, [field]: value } : s)
  }));
}
```

**Mutation**:
```typescript
const qc = useQueryClient();
const mutation = useMutation({
  mutationFn: () => abTests.create({
    name: form.name.trim(),
    traffic_split: form.splits.map(s => ({ variant: s.variant, weight: s.weight })),
    // model은 lib/api.ts의 ABTestTarget에 있으나 백엔드에서는 TrafficSplit.Model에 있음.
    // CreateABTestPayload를 그대로 사용해 target.model에 splits[0].model을 넣거나,
    // target을 { model: "", sample_rate: form.sample_rate }로 설정.
    target: { model: form.splits[0]?.model ?? "", sample_rate: form.sample_rate },
    min_samples: form.min_samples,
    confidence_level: form.confidence_level,
    ...(form.start_at ? { start_at: new Date(form.start_at).toISOString() } : {}),
    ...(form.end_at   ? { end_at:   new Date(form.end_at).toISOString()   } : {}),
  }),
  onSuccess: () => {
    qc.invalidateQueries({ queryKey: ["ab-tests"] });
    onCreated();
  },
  onError: (e: Error) => setError(e.message),
});
```

**폼 레이아웃** (모달 내부):
```
[Name *]
──── Traffic Split ────
  Variant    Model                 Weight (%)   [×]
  [control ] [gpt-4o           ]  [50       ]
  [treatment] [claude-3-5-sonnet] [50       ]
  [+ Add Variant]
  Warning: weights must sum to 100 (if invalid)
──── Options ────
  Min Samples [1000]   Confidence Level [0.95]   Sample Rate [1.0]
  Start At [datetime-local]   End At [datetime-local]
──── Footer ────
  [Cancel]  [Create Experiment]
```

### 3.4 ResultsPanel

**역할**: 선택된 실험의 variant별 성능 지표를 인라인 패널로 표시.

**Props**:
```typescript
function ResultsPanel({ test, onClose }: {
  test: ABTest;
  onClose: () => void;
})
```

**데이터 페칭**:
```typescript
const { data: rawResults, isLoading, error } = useQuery({
  queryKey: ["ab-test-results", test.id],
  queryFn: () => abTests.results(test.id),
  enabled: true,
});

// 로컬 타입으로 캐스팅
const analysis = rawResults as unknown as AnalysisResult | undefined;
```

**렌더링 조건**:
- `isLoading` → `"Loading results…"` 텍스트
- `error` → `"결과를 불러오지 못했습니다: {error.message}"` 에러 메시지
- `!analysis?.results || Object.keys(analysis.results).length === 0` → `"No results yet (0 samples)"` 빈 상태

**메인 테이블 (variant별 지표)**:

```
| Variant | Model | Samples | Latency P95 (ms) | Avg Cost/Req ($) | Error Rate (%) |
|---------|-------|---------|-----------------|------------------|----------------|
| control | gpt-4o | 1,234  | 432.1           | $0.0023          | 0.81%          |
```

- `samples === 0` 인 경우 해당 행에 `"No data yet"` 표시
- `avg_cost_per_request` 표시: `$${v.avg_cost_per_request.toFixed(4)}`
- `error_rate` 표시: `${(v.error_rate * 100).toFixed(2)}%`

**통계적 유의성 섹션** (별도 테이블, `analysis.statistical_significance` 있을 때만 표시):

```
Statistical Significance
| Metric | P-Value | Significant | Improvement |
|--------|---------|-------------|-------------|
| latency_p95_ms | 0.023 | Yes | -12.3% |
```

**Recommendation 섹션** (`analysis.recommendation` 비어있지 않을 때만 표시):
```tsx
{analysis.recommendation && (
  <div className="mt-3 p-3 bg-blue-50 rounded text-sm text-blue-800">
    <span className="font-medium">Recommendation: </span>
    {analysis.recommendation}
  </div>
)}
```

**Winner 표시** (`analysis.winner` 있을 때):
```tsx
{analysis.winner && (
  <p className="text-xs text-green-700 font-medium">
    Winner: {analysis.winner}
  </p>
)}
```

### 3.5 PromoteDialog

**역할**: 실험의 승자 variant를 선택하고 `/promote` API를 호출하는 확인 모달.

**Props**:
```typescript
function PromoteDialog({ test, onClose }: {
  test: ABTest;
  onClose: () => void;
})
```

**내부 상태**:
```typescript
const [winner, setWinner] = useState<string>("");
const [error, setError] = useState("");
```

**초기화**: `useState`로 `winner`를 `""`으로 초기화한다. 기존 winner가 있는 경우(`(test as any).winner`) `useEffect` 없이 select의 defaultValue로 처리하거나 초기값으로 설정한다.

**Mutation**:
```typescript
const qc = useQueryClient();
const mutation = useMutation({
  mutationFn: () => abTests.promote(test.id, winner),
  onSuccess: () => {
    qc.invalidateQueries({ queryKey: ["ab-tests"] });
    onClose();
  },
  onError: (e: Error) => setError(e.message),
});
```

**렌더링**:
```tsx
<div className="fixed inset-0 bg-black/40 flex items-center justify-center z-50">
  <div className="bg-white rounded-xl shadow-xl w-full max-w-sm p-6 space-y-4">
    <h2 className="text-lg font-semibold text-slate-900">Promote Winner</h2>
    <p className="text-sm text-slate-500">
      Select the winning variant for <strong>{test.name}</strong>.
      The experiment will be marked as completed.
    </p>

    {error && <p className="text-sm text-red-600 bg-red-50 rounded p-2">{error}</p>}

    <div>
      <label className="text-xs font-medium text-slate-600">Winner Variant</label>
      <select
        value={winner}
        onChange={(e) => setWinner(e.target.value)}
        className="mt-1 block w-full border border-slate-300 rounded-lg px-3 py-2 text-sm"
      >
        <option value="">— Select variant —</option>
        {(test.traffic_split as unknown as TrafficSplitWithModel[]).map((s) => (
          <option key={s.variant} value={s.variant}>{s.variant}</option>
        ))}
      </select>
    </div>

    <div className="flex justify-end gap-2 pt-2">
      <button onClick={onClose} className="px-4 py-2 text-sm text-slate-600 hover:bg-slate-100 rounded-lg">
        Cancel
      </button>
      <button
        disabled={!winner || mutation.isPending}
        onClick={() => mutation.mutate()}
        className="px-4 py-2 text-sm bg-slate-900 text-white rounded-lg disabled:opacity-50"
      >
        {mutation.isPending ? "Promoting…" : "Promote"}
      </button>
    </div>
  </div>
</div>
```

## 5. Form Validation Logic

### weight 합산 100 검증 (CreateDialog)

```typescript
// 검증 상태 파생
const totalWeight = form.splits.reduce((sum, s) => sum + (s.weight || 0), 0);
const weightValid = totalWeight === 100;

// Submit 버튼 disabled 조건
const canSubmit =
  form.name.trim().length > 0 &&
  form.splits.length >= 2 &&
  form.splits.every(s => s.variant.trim().length > 0 && s.model.trim().length > 0) &&
  weightValid &&
  !mutation.isPending;
```

### weight 입력 처리

- `<input type="number" min={0} max={100}>` 사용
- 입력값은 항상 `Number(e.target.value)` 변환 후 저장
- `NaN` 보호: `isNaN(parsed) ? 0 : parsed`

### datetime 입력 처리

- `<input type="datetime-local">` 사용
- 빈 문자열이면 payload에서 제외 (`...(form.start_at ? { start_at: ... } : {})`)
- ISO 변환: `new Date(form.start_at).toISOString()`

## 6. Error and Loading State Handling

### 전역 (ABTestsPage)

| 상황 | 처리 |
|---|---|
| `isLoading === true` | 테이블 대신 `<p className="text-sm text-slate-400">Loading…</p>` |
| `testList.length === 0 && !isLoading` | Empty state UI (메시지 + Create 버튼) |
| 상태 전환 실패 | `transitionError` state에 저장 → 목록 상단 빨간 텍스트로 표시 |
| `useQuery` error | `error?.message`를 빨간 텍스트로 표시 |

### ResultsPanel

| 상황 | 처리 |
|---|---|
| `isLoading` | `"Loading results…"` |
| `error` | `"결과를 불러오지 못했습니다: {message}"` |
| `results` 빈 객체 또는 undefined | `"No results yet (experiment has not started or has 0 samples)"` |
| variant의 `samples === 0` | 해당 행에 이탈리체로 `"No data yet"` 표시 |
| `statistical_significance` 없음 | 통계 섹션 미표시 (graceful hidden) |
| `recommendation` 빈 문자열 또는 없음 | Recommendation 섹션 미표시 |

### CreateDialog

| 상황 | 처리 |
|---|---|
| weight 합산 ≠ 100 | Submit disabled + 빨간 경고 텍스트 |
| API 422 응답 | `error` state에 저장 → 모달 상단 빨간 텍스트, 모달 유지 |
| `mutation.isPending` | Submit 버튼 "Creating…" 텍스트, disabled |

### PromoteDialog

| 상황 | 처리 |
|---|---|
| winner 미선택 | Promote 버튼 disabled |
| API 에러 | `error` state → 모달 내 빨간 텍스트, 모달 유지 |
| `mutation.isPending` | "Promoting…" 텍스트, disabled |

## 10. Edge Cases 처리 정리

| 케이스 | 처리 방식 |
|---|---|
| 실험 목록 비어있음 | Empty state UI (메시지 + Create 버튼) |
| weight 합산 != 100 | Submit disabled + 빨간 경고 문구 |
| `results.samples === 0` | 테이블 셀에 이탤릭체 "No data yet" |
| `recommendation` 없음/빈 문자열 | 해당 섹션 미표시 (조건부 렌더링) |
| `statistical_significance` 없음 | 통계 섹션 미표시 |
| `completed`/`stopped` 상태 | Start/Pause/Stop 버튼 미표시; Results만 표시 |
| Promote winner 이미 설정됨 | select 초기값으로 기존 winner 표시 후 재선택 가능 |
| `created_at` 없음 (lib/api.ts 미선언) | `"—"` 폴백 표시 |
| `lib/api.ts` 401 응답 | `apiFetch` 내부에서 `/login` 리다이렉트 처리 (기존 동작) |
| ATL-290과 Sidebar.tsx 충돌 | 해당 PR rebase 후 항목만 삽입 |
