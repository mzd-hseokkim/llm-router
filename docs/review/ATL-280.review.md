## Code Quality Findings

### Critical (Fixed)

- **`routingRules.dryRun` wrong path** (`admin-ui/lib/api.ts`)
  Client used `/routing/rules/dry-run` but backend registers the endpoint as `/admin/routing/test`.
  **Fixed**: Changed to `/routing/test`. Would have caused 404 at runtime.

### Warnings

- **`RoutingConfig.providers` and `.rules` typed as `unknown[]`**
  The `routing.get()` / `routing.update()` use a loose `RoutingConfig` interface with `providers: unknown[]` and `rules: unknown[]`. Acceptable for now since the routing config envelope differs from the routing rules model, but consumers will need type assertions.

- **`budgets.reset` return type mismatch**
  Backend returns HTTP 501 with `{ error: "..." }`. Since `apiFetch` throws on non-2xx responses, the declared return type `Promise<{ error: string }>` is unreachable — the function always throws. Low risk (501 means it never succeeds), but the return type is misleading.

- **`abTests.results` typed as `Record<string, unknown>`**
  The `/ab-tests/{id}/results` response is the output of a statistical analysis function whose shape isn't fixed. Appropriate placeholder, but consumers will need type assertions.

### Info

- **`securityEvents` in `auditLogs` only exposes a subset of filter params**
  `auditLogs.list` exposes all 7 params; `auditLogs.securityEvents` only exposes 4 (`from`, `to`, `limit`, `page`). The backend likely accepts the full set for both. Not blocking, but may limit future UI capabilities.

- **`ShowbackReport.breakdown` typed as `unknown[]`**
  The backend didn't expose a concrete type for the showback breakdown. Fine as a placeholder.

- **DELETE methods use raw `fetch` (not `apiFetch`)**
  `routingRules.delete` and `mcp.servers.delete` use raw `fetch(...)` consistent with the existing pattern for `keys.deactivate`, `providers.delete`, and `providerKeys.delete`. Intentional pattern — DELETE responses don't return a body.
