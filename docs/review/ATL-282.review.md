## Code Quality Findings

### ⚠️ Warnings

**W1: 중복 쿼리 갱신 — `page.tsx:197`**
```typescript
function handleSearch() {
  if (!entityId || !isValidUUID) return;
  setSearchKey({ type: entityType, id: entityId });
  qc.invalidateQueries({ queryKey: ["budgets"] }); // 불필요
}
```
`setSearchKey`로 queryKey가 변경되면 React Query가 자동으로 새 데이터를 fetch하므로 `invalidateQueries` 호출이 중복이다. 이 경우 fetch가 두 번 발생할 수 있다. 제거해도 무방하다.

**W2: 수정 진입점 불명확 — `page.tsx:293-299`**
```typescript
<button onClick={() => openEdit(b)} className="font-medium text-brand-600 hover:underline">
  {b.period}
</button>
```
테이블의 "기간" 셀을 클릭해야 수정 모달이 열리는 구조인데, 시각적 단서가 underline+색상뿐이라 발견하기 어렵다. 운영 환경에서 사용자 혼란이 생길 수 있다. 별도 편집 아이콘(✏️ 버튼) 추가를 권장하나, 기획 문서에 명시적 요구가 없으므로 Warning 수준.

### ℹ️ Info

**I1: 타입 중복 어노테이션 — `page.tsx:290`**
```typescript
{budgetList.map((b: Budget) => (
```
`budgetList`가 이미 `Budget[]`로 추론되므로 `(b: Budget)` 어노테이션은 불필요. 기능에는 무관하다.

**I2: `pct` 변수 명명 — `page.tsx:22-23`**
```typescript
const pct = Math.min((budget.current_spend / limit) * 100, 110);
const displayPct = Math.round((budget.current_spend / limit) * 100);
```
두 변수가 다른 목적(바 너비 vs 표시 텍스트)으로 분리된 것은 의도적이고 올바르다. 그러나 `pct`는 실제로 getBarColor 색상 판별에만 사용되며 `Math.min(..., 110)`의 의미가 명확하지 않다. 110% 캡은 `Math.min(pct, 100)%`로 바 너비가 제한되므로 실제로는 `100`이어도 동일하게 동작한다.

## Positive Notes

- **UUID 유효성 검사 일관성**: 검색 폼(page.tsx:183-185)과 모달(page.tsx:65-67) 양쪽에 동일한 정규식 검증이 적용되어 있다.
- **null-safe 렌더링**: `soft_limit_usd`, `hard_limit_usd`가 `null`인 경우 "미설정"으로 처리(page.tsx:301-304)하고, Progress Bar에서는 "한도 없음"으로 처리(page.tsx:18-20)하는 등 null 핸들링이 일관되다.
- **Progress Bar 색상 규칙**: 기획서 스펙(0~60% green, 60~80% yellow, 80~100% orange, 100%+ red)을 정확히 구현했다.
- **컴포넌트 분리**: `SpendBar`, `BudgetModal` 을 별도 컴포넌트로 분리하여 가독성이 좋다.
- **엣지 케이스 처리**: 빈 목록 상태(page.tsx:332-336), API 오류(page.tsx:272-274), 로딩 상태(page.tsx:269-270)를 모두 처리했다.
- **upsert 패턴 정확 반영**: 백엔드 API 제약(POST only, upsert)에 맞게 생성/수정 모두 `budgets.create`를 사용하는 설계가 올바르다.
- **리셋 버튼 비활성화**: 501 제약사항을 UI에서 정확히 반영하여 disabled + 안내 툴팁으로 처리했다.
