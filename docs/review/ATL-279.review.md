## Code Quality Findings

### Warnings

**`internal/provider/openai/embed.go:23-32` — 불필요한 `json.RawMessage` 사용**

```go
// 현재
modelJSON, _ := json.Marshal(model)
reqBody := struct {
    Model json.RawMessage `json:"model"`
    ...
}{
    Model: modelJSON,
}

// 권장
reqBody := struct {
    Model string `json:"model"`
    ...
}{
    Model: model,
}
```

`model`은 이미 bare string이므로 `json.RawMessage`로 감쌀 이유가 없다. 동작은 동일하지만 코드가 불필요하게 복잡하다. `json.Marshal(model)`의 오류는 무시(`_`)되어 있는데, 이 구조에서 오류는 사실상 불가능하므로 실질적 위험은 없다.

### Info

**`internal/cache/exact/key_test.go:TestBuildEmbeddingKey_WhitespaceNormalization` — 동일 입력으로 테스트**

```go
r1 := &types.EmbeddingRequest{Model: "m", Input: []byte(`"a"`)}
r2 := &types.EmbeddingRequest{Model: "m", Input: []byte(`"a"`)}
// 두 입력이 완전히 동일 — whitespace 차이를 검증하지 않음
```

설계 의도는 `{"input":"a"}` vs `{"input": "a"}` 같이 whitespace 차이가 있는 입력이 동일 키를 반환함을 검증하는 것이었다. 현재 테스트는 동일한 입력이 동일 키를 반환함만 검증하므로 whitespace 정규화 기능이 실제로 테스트되지 않는다. (기능 자체는 정상 구현됨)

**`internal/provider/cohere/embed.go:53` — `input_type` 하드코딩**

```go
InputType: "search_document",
```

`"search_document"`는 일반 RAG 검색에 적합하지만, 쿼리 임베딩(`"search_query"`)과 구분이 필요한 Cohere 모델에서는 정확도 차이가 발생할 수 있다. 설계 문서에서도 `[ ]`로 미해결 항목으로 명시됨. 현재 버전의 범위 내에서는 허용 가능하다.

## Positive Notes

- **설계 충실도**: 설계 문서의 모든 항목이 정확히 구현됨. 파일 구조, 시그니처, 동작 모두 설계와 일치.
- **캐시 오류 비전파**: `StoreEmbedding` / `GetEmbedding` 실패가 핸들러 흐름을 중단시키지 않는 설계 원칙이 코드에 정확히 반영됨.
- **기존 패턴 일관성**: `StoreEmbedding`의 maxSize 체크 → gzip 압축 순서가 기존 `Store` 패턴과 완전히 일치.
- **테스트 품질**: mock HTTP 서버를 사용한 실제 HTTP 요청/응답 기반 테스트가 모든 provider에 대해 구현됨.
- **보안**: keyFunc(ctx) 패턴으로 API 키가 핸들러에 전달되지 않으며, 캐시 키에 인증 정보가 포함되지 않음.
- **`IsEmbeddingCacheable(nil)` 안전성**: nil 맵 접근 시 Go 빈 문자열 반환 특성을 이용한 안전한 설계.
- **Cohere baseURL 확인**: `defaultBaseURL = "https://api.cohere.com/v2"`이므로 `a.baseURL+"/embed"` = `https://api.cohere.com/v2/embed` — 설계 스펙과 일치.
