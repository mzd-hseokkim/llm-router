## Code Quality Findings

### Critical
없음.

### Warnings

**W1. `prompt_store.go:ListPromptsPage` — 포지셔널 파라미터 계산 혼란 (가독성)**

```go
args = append(args, limit, offset)
limitPos := len(args) - 1   // after appending BOTH; works but confusing
offsetPos := len(args)
query += fmt.Sprintf(" LIMIT $%d OFFSET $%d", limitPos, offsetPos)
```

두 원소를 한번에 append한 후 `len(args)-1`로 limit 위치를 계산하는 방식이 작동은 하지만,
append 순서가 바뀌거나 원소가 추가되면 조용히 깨진다.
더 명확한 방식: `len(args)+1`, `len(args)+2`를 append 전에 계산.

**W2. `admin_prompts.go` — DB 에러 메시지 직접 노출**

```go
writeError(w, http.StatusInternalServerError, err.Error(), "api_error", "")
```

`err.Error()`를 클라이언트에 그대로 반환. SQL 오류, 테이블명 등 내부 구현 정보가 유출될 수 있음.
단, 이 패턴은 이미 기존 코드에 존재하며 ATL-469에서 신규 도입한 것이 아님.

**W3. `admin_keys.go`/`main.go` — 인터페이스에서 구체 타입으로 변경**

```go
// Before: func registerAdminRoutes(r chi.Router, store auth.Store, ...)
// After:  func registerAdminRoutes(r chi.Router, store *pgstore.VirtualKeyStore, ...)
```

`registerAdminRoutes`의 store 파라미터가 인터페이스(`auth.Store`)에서 구체 타입(`*pgstore.VirtualKeyStore`)으로
변경됨. 단위 테스트 시 mock 적용이 어려워지고 DI 원칙 위반. 더 나은 방안: `auth.Store`를 확장하거나
`keyStore` 인터페이스를 별도 파일에서 선언 후 wiring.

단, 현재 구조에서 `auth.Store`는 외부 패키지 인터페이스라 바로 확장하기 어렵고, 이 함수의 테스트가
없으므로 실질적 리스크는 낮음.

### Info

**I1. `api.ts` — `page=0` 또는 `limit=0` 시 URL 파라미터 누락**

```typescript
if (params?.page) q.set("page", String(params.page));  // 0은 falsy
```

페이지 번호 0은 유효하지 않으므로 실질적 버그는 없지만, `params.page != null && params.page > 0`이
더 명시적.

**I2. `CountKeys()` — is_active 필터 없음**

`CountKeys()`는 모든 키(비활성 포함) 수를 반환. `total` 카운트와 실제 보이는 키 수가 불일치할 수 있어
UX상 약간의 혼란 가능성. 현재는 목록에도 비활성 키가 표시되므로 일관성은 있음.

**I3. `admin_logs.go` — limit 기본값 이중 계산**

`page` 분기 내부와 응답 직전에 `f.Limit <= 0 → 100` 기본값 로직이 중복. Store에서도 동일하게
처리하므로 3중 기본값 적용. 기능상 문제 없으나 일관성 개선 여지 있음.
