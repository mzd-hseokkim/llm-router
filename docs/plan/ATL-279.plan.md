# ATL-279: Embeddings API 구현 (/v1/embeddings)

**작성일**: 2026-02-23
**우선순위**: 주요 (Major)
**브랜치**: `feature/ATL-279`

---

## Background

`/v1/embeddings` 엔드포인트는 라우터에 이미 등록되어 있고 (`router/router.go:109`), `EmbeddingsHandler` 골격도 존재하지만 실제 provider 호출 로직이 없다:

```go
// TODO: delegate to provider embedding (task 03)
writeJSON(w, http.StatusOK, types.EmbeddingResponse{Object: "list", Data: []types.Embedding{}, ...})
```

현재 핸들러는 요청 파싱과 provider 유효성 검사만 수행하고, 항상 빈 벡터 배열을 반환한다.

## Scope

### In Scope
- `EmbeddingProvider` 선택적 인터페이스 정의
- OpenAI, Gemini, Cohere 어댑터에 `Embed()` 구현
- 임베딩 결과 캐싱 (`exact.Cache` 확장, Redis 기반)
- 비용 추적 (기존 `cost.Calculator` 활용, embedding 단가 추가)
- `EmbeddingsHandler` 완성 (provider 호출 + 캐시 + 비용)
- Router 및 DI 배선 업데이트
- 유닛 테스트, E2E 테스트

### Out of Scope
- Anthropic, Azure, Bedrock, Mistral, Grok, 자체호스팅 임베딩 (해당 provider에 임베딩 API가 없거나 OpenAI-compatible 프록시로 커버됨)
- 의미론적 유사도 캐시 (semantic cache)
- 스트리밍 임베딩 (OpenAI 표준 없음)
- 임베딩 요청에 대한 Guardrail 적용 (메시지 구조가 아님)

## Technical Design

### 1. `EmbeddingProvider` 인터페이스 (선택적)

**파일**: `internal/provider/interface.go`

```go
// EmbeddingProvider is implemented by adapters that support the embeddings API.
// Not all providers support embeddings; handlers should type-assert before use.
type EmbeddingProvider interface {
    Embed(ctx context.Context, model string, req *types.EmbeddingRequest) (*types.EmbeddingResponse, error)
}
```

기존 `Provider` 인터페이스는 변경하지 않음 — 선택적 인터페이스로 분리한다.
핸들러에서 `p, ok := prov.(provider.EmbeddingProvider)` 형태로 타입 어설션.

---

### 2. Provider별 `Embed()` 구현

#### 2a. OpenAI (`internal/provider/openai/`)
- 엔드포인트: `POST {baseURL}/embeddings`
- 요청 형식: OpenAI 네이티브 — 모델 prefix(`openai/`) 제거 후 그대로 전달
- 응답: 그대로 파싱

#### 2b. Gemini (`internal/provider/gemini/`)
- 단일 입력: `POST /v1beta/models/{model}:embedContent?key={apiKey}`
  ```json
  {"content": {"parts": [{"text": "..."}]}}
  ```
- 배치 입력: `POST /v1beta/models/{model}:batchEmbedContents?key={apiKey}`
  ```json
  {"requests": [{"content": {"parts": [{"text": "..."}]}}]}
  ```
- `req.Input`을 파싱해 string이면 단일, `[]string`이면 배치 API 사용
- 응답: `{"embedding": {"values": [...]}}` 또는 `{"embeddings": [{"values": [...]}]}`

#### 2c. Cohere (`internal/provider/cohere/`)
- 엔드포인트: `POST {baseURL}/embed` (v2 API)
- 요청:
  ```json
  {
    "model": "embed-english-v3.0",
    "texts": ["..."],
    "input_type": "search_document",
    "embedding_types": ["float"]
  }
  ```
- 응답: `{"embeddings": {"float": [[...]]}, "meta": {"billed_units": {"input_tokens": N}}}`

### 3. 임베딩 캐시 확장

**파일**: `internal/cache/exact/cache.go` 및 `key.go` 확장

새 타입 추가 (기존 `Entry`, `BuildKey` 등 변경 없음):

```go
// EmbeddingEntry는 임베딩 응답 캐시 항목.
type EmbeddingEntry struct {
    Response    *types.EmbeddingResponse `json:"response"`
    CreatedAt   int64                    `json:"created_at"`
    Model       string                   `json:"model"`
    InputTokens int                      `json:"input_tokens"`
    CostUSD     float64                  `json:"cost_usd"`
}
```

새 메서드 추가:
- `GetEmbedding(ctx, key string) (*EmbeddingEntry, error)` — key prefix: `cache:emb:`
- `StoreEmbedding(ctx, key string, resp *types.EmbeddingResponse, costUSD float64, ttl time.Duration) error`

키 생성:
```go
// BuildEmbeddingKey derives SHA-256 key from (model, encoding_format, input).
func BuildEmbeddingKey(req *types.EmbeddingRequest) string
```

캐시 가능 조건: `Cache-Control: no-cache`와 `X-Gateway-No-Cache: true`가 없으면 항상 캐시 (임베딩은 결정론적).

### 4. `EmbeddingsHandler` 완성

**파일**: `internal/gateway/handler/embeddings.go`

의존성 추가:
```go
type EmbeddingsHandler struct {
    registry  *provider.Registry
    cache     *exact.Cache    // optional, nil = no caching
    costCalc  *cost.Calculator // optional, nil = no cost tracking
    logger    *slog.Logger
}
```

처리 흐름:
1. 요청 파싱 + 유효성 검사 (기존 유지)
2. `parsed := parseModel(req.Model)` → provider 조회
3. `p.(provider.EmbeddingProvider)` 타입 어설션 → 실패 시 `400`
4. 캐시 조회 (`BuildEmbeddingKey`) → 히트 시 `X-Cache: HIT` 헤더 + 캐시 응답 반환
5. `p.Embed(ctx, parsed.Model, &req)` 호출
6. 비용 계산: `costCalc.Calculate(req.Model, usage.TotalTokens, 0)`
7. 캐시 저장 (비동기 불필요 — 동기 ok)
8. Telemetry: `SetTokens`, `SetModel`
9. JSON 응답 반환

## Risks & Edge Cases

| 위험 | 대응 |
|------|------|
| `Input` json.RawMessage가 따옴표 인코딩이 다를 수 있어 캐시 키 충돌 | `json.Unmarshal` → 정규화 후 재직렬화하여 키 생성 |
| Gemini 임베딩 모델 이름에 `models/` prefix 필요 여부 | API 응답 테스트로 확인, `transform.go`에서 처리 |
| Cohere `input_type` 파라미터 — caller가 목적을 모를 수 있음 | 기본값 `"search_document"` 사용, 필요 시 `EmbeddingRequest` 확장 고려 |
| 임베딩 배열이 매우 클 수 있어 캐시 max size 초과 가능 | 기존 `ErrResponseTooLarge` 로직 재사용; 초과 시 캐시 생략 |
| `exact.Cache` 확장 시 기존 채팅 캐시 키와 충돌 | key prefix를 `cache:emb:` 로 분리 (기존: `cache:exact:`) |
