## 3. Architecture Decision: Vitest 선택 이유

### 결정: Vitest (Jest 대신)

**배경**: Next.js 15는 Vite 기반 테스트 도구와 잘 맞고, React 19의 실험적 기능과의 호환성이 중요하다.

| 기준 | Vitest | Jest |
|---|---|---|
| Vite/Next.js 15 호환 | ✅ 네이티브 | ⚠️ 설정 복잡 |
| React 19 지원 | ✅ | ⚠️ 일부 이슈 |
| TypeScript 설정 | ✅ 간단 | ⚠️ 별도 transform 필요 |
| 속도 | ✅ 빠름 | 보통 |
| API 호환성 | Jest와 동일한 API | - |

**전략**:
- `vitest` + `@vitejs/plugin-react` + `@testing-library/react` + `jsdom`
- `@testing-library/user-event`로 사용자 인터랙션 테스트
- `vitest.config.ts`에서 Next.js path alias(`@/*`) 해소

### 테스트 환경 설정 접근

Next.js 15의 App Router 컴포넌트는 Server Component / Client Component가 혼재한다.
- **Server Components**: 직접 테스트 어렵고 범위 밖 (smoke test 제외)
- **Client Components** (`'use client'`): testing-library로 직접 테스트 가능
- **Page Components**: 대부분 data fetch를 포함하므로 fetch/router mock 필요

## 6. Mock 전략

### next/navigation Mock (vitest.setup.ts 또는 각 test 파일)
```ts
vi.mock('next/navigation', () => ({
  usePathname: () => '/',
  useRouter: () => ({ push: vi.fn(), replace: vi.fn() }),
  useSearchParams: () => new URLSearchParams(),
}))
```

### fetch Mock
```ts
global.fetch = vi.fn().mockResolvedValue({
  ok: true,
  json: async () => ({}),
})
```

### QueryClient 래핑 헬퍼
```ts
function renderWithQueryClient(ui: React.ReactElement) {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } },
  })
  return render(<QueryClientProvider client={queryClient}>{ui}</QueryClientProvider>)
}
```

## 7. Risks & Mitigations

| 리스크 | 가능성 | 영향 | 대응 |
|---|---|---|---|
| React 19 + testing-library 호환 이슈 | 중간 | 중간 | `@testing-library/react@latest` 사용, act() 경고 무시 설정 |
| Next.js App Router Server Component 테스트 불가 | 높음 | 낮음 | Client Component만 테스트, Server Component는 smoke test 제외 |
| path alias `@/*` 해소 실패 | 낮음 | 중간 | vitest.config.ts의 `resolve.alias` 명시적 설정 |
| `next/navigation` mock 불완전 | 중간 | 중간 | 각 테스트에서 필요한 부분만 mock, 전역 mock 파일로 관리 |

---

## 8. Edge Cases

1. **Server Component + Client Component 혼재**: `'use client'` 없는 컴포넌트는 테스트 대상 제외
2. **동적 import**: `next/dynamic`으로 lazy load된 컴포넌트 → `vi.mock` 필요 시 처리
3. **httpOnly 쿠키 기반 인증**: API 클라이언트는 직접 테스트하지 않고 mock으로 대체
4. **recharts 렌더링**: jsdom에서 SVG 렌더링 제한 → 차트 컴포넌트는 mock 처리
