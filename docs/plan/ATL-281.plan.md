## 1. Background

LLM Router Admin UI에 라우팅 규칙 관리 페이지(`/routing`)가 없어 운영 불가 상태다.
백엔드 API(`/admin/routing`, `/admin/routing/rules/*`)와 `lib/api.ts`의 `routing`, `routingRules` 객체는 이미 완성되어 있으며, UI만 없는 상황이다.

### 현재 상태

| 컴포넌트 | 위치 | 상태 |
|---|---|---|
| Backend API | `/admin/routing`, `/admin/routing/rules/*` | ✅ 완성 |
| API 클라이언트 | `lib/api.ts` — `routing`, `routingRules` 객체 | ✅ 완성 |
| Sidebar 링크 | `components/Sidebar.tsx` — Routing 항목 | ✅ 이미 추가됨 |
| 라우팅 페이지 | `app/(admin)/routing/page.tsx` | ⚠️ 플레이스홀더만 존재 |

## 2. Backend API Contract

### 엔드포인트 목록

| Method | Endpoint | 설명 |
|---|---|---|
| `GET` | `/admin/routing` | 전체 라우팅 설정 조회 → `RoutingConfig` |
| `PUT` | `/admin/routing` | 전체 라우팅 설정 업데이트 → `{ status: "updated" }` |
| `POST` | `/admin/routing/reload` | 설정 재적용 → `{ status: string }` |
| `GET` | `/admin/routing/rules` | 규칙 목록 → `{ data: RoutingRule[] }` |
| `POST` | `/admin/routing/rules` | 규칙 생성 → `RoutingRule` |
| `PUT` | `/admin/routing/rules/{id}` | 규칙 수정 → `RoutingRule` |
| `DELETE` | `/admin/routing/rules/{id}` | 규칙 삭제 → 204 No Content |
| `POST` | `/admin/routing/test` | Dry-run 테스트 → `DryRunResult` |

### 핵심 타입 (Backend `internal/gateway/types/routing.go`)

```go
type Strategy string

const (
  StrategyDirect    Strategy = "direct"        // 단일 타겟
  StrategyWeighted  Strategy = "weighted"      // 가중치 랜덤
  StrategyLeastCost Strategy = "least_cost"    // 가장 저렴한 타겟
  StrategyFailover  Strategy = "failover"      // 순차적 폴백
  StrategyQuality   Strategy = "quality_first" // 품질 우선
)

type RuleTarget struct {
  Provider string `json:"provider"`
  Model    string `json:"model"`
  Weight   int    `json:"weight,omitempty"`
}

type RouteMatch struct {
  Model            string            `json:"model,omitempty"`
  ModelPrefix      string            `json:"model_prefix,omitempty"`
  ModelRegex       string            `json:"model_regex,omitempty"`
  KeyID            *uuid.UUID        `json:"key_id,omitempty"`
  UserID           *uuid.UUID        `json:"user_id,omitempty"`
  TeamID           *uuid.UUID        `json:"team_id,omitempty"`
  OrgID            *uuid.UUID        `json:"org_id,omitempty"`
  Metadata         map[string]string `json:"metadata,omitempty"`
  MinContextTokens int               `json:"min_context_tokens,omitempty"`
  MaxContextTokens int               `json:"max_context_tokens,omitempty"`
  HasTools         bool              `json:"has_tools,omitempty"`
}

type RouteRule struct {
  ID        uuid.UUID    `json:"id"`
  Name      string       `json:"name"`
  Priority  int          `json:"priority"`
  Enabled   bool         `json:"enabled"`
  Match     RouteMatch   `json:"match"`
  Strategy  Strategy     `json:"strategy"`
  Targets   []RuleTarget `json:"targets"`
  CreatedAt time.Time    `json:"created_at"`
  UpdatedAt time.Time    `json:"updated_at"`
}
```

### `lib/api.ts` 타입 (이미 완성 — 수정 불필요)

```typescript
export interface RoutingRule {
  id: string;
  name: string;
  priority: number;
  enabled: boolean;
  match: RoutingRuleMatch;
  strategy: string;
  targets: RoutingRuleTarget[];
  created_at: string;
  updated_at: string;
}

export interface RoutingRuleMatch {
  model?: string;
  model_prefix?: string;
  model_regex?: string;
  key_id?: string;
  user_id?: string;
  team_id?: string;
  org_id?: string;
  metadata?: Record<string, string>;
  min_context_tokens?: number;
  max_context_tokens?: number;
  has_tools?: boolean;
}

export interface RoutingRuleTarget {
  provider: string;
  model: string;
  weight?: number;
}

export interface CreateRoutingRulePayload {
  name: string;
  priority?: number;
  enabled?: boolean;
  match?: RoutingRuleMatch;
  strategy: string;
  targets: RoutingRuleTarget[];
}

export interface DryRunPayload {
  model: string;
  metadata?: Record<string, string>;
  messages?: Array<{ role: string; content: string }>;
}

export const routingRules = {
  list: () => apiFetch<{ data: RoutingRule[] }>("/routing/rules").then(r => r.data),
  create: (payload: CreateRoutingRulePayload) => apiFetch<RoutingRule>("/routing/rules", ...),
  update: (id: string, payload: CreateRoutingRulePayload) => apiFetch<RoutingRule>(`/routing/rules/${id}`, ...),
  delete: (id: string) => fetch(`${BASE}/routing/rules/${id}`, { method: "DELETE" }),
  reload: () => apiFetch<{ status: string }>("/routing/rules/reload", ...),
  dryRun: (payload: DryRunPayload) => apiFetch<DryRunResult>("/routing/test", ...),
};
```

## 4. Acceptance Criteria

### AC-1: 규칙 목록 표시
- **Given** `/routing` 페이지에 접속했을 때
- **When** 페이지가 로드되면
- **Then** 라우팅 규칙 목록이 priority 오름차순으로 테이블에 표시된다
- **And** 각 규칙의 이름, 우선순위, 활성 여부, strategy, match 요약, targets 수가 표시된다

### AC-2: 규칙 생성
- **Given** "+ Add Rule" 버튼을 클릭했을 때
- **When** 이름, strategy, 최소 1개의 target(provider+model)을 입력하고 저장하면
- **Then** 새 규칙이 목록에 추가되고 모달이 닫힌다

### AC-3: 규칙 편집
- **Given** 규칙 행의 "Edit" 버튼을 클릭했을 때
- **When** 기존 값이 채워진 편집 모달이 열리고 수정 후 저장하면
- **Then** 변경사항이 반영되고 목록이 갱신된다

### AC-4: 규칙 삭제
- **Given** 규칙 행의 "Delete" 버튼을 클릭했을 때
- **When** 확인 dialog에서 "Delete"를 누르면
- **Then** 해당 규칙이 목록에서 제거된다

### AC-5: 규칙 활성/비활성 토글
- **Given** 규칙의 enabled 토글을 클릭했을 때
- **When** PUT `/routing/rules/{id}` 요청이 성공하면
- **Then** 목록의 활성 상태 표시가 즉시 갱신된다

### AC-6: DryRun 테스트
- **Given** DryRun 패널에서 model 이름을 입력하고 "Test" 버튼을 클릭했을 때
- **When** POST `/routing/test` 응답이 도착하면
- **Then** 매칭된 규칙 이름, strategy, 선택된 targets가 표시된다
- **And** 매칭된 규칙이 없으면 "No matching rule — default routing applied" 메시지가 표시된다

### AC-7: Reload 버튼
- **Given** 규칙 목록 헤더의 "Reload" 버튼을 클릭했을 때
- **When** POST `/routing/reload` 요청이 성공하면
- **Then** "Configuration reloaded" 상태 메시지가 잠깐 표시된다

### 규칙 테이블 컬럼

| 컬럼 | 설명 |
|---|---|
| Priority | 숫자 (오름차순 정렬) |
| Name | 규칙 이름 |
| Enabled | 토글 배지 (초록/회색) |
| Match | 조건 요약 (예: "model: gpt-4*", "context ≤ 8k") |

| Targets | 타겟 수 (예: "2 targets") |
| Actions | Edit / Delete 버튼 |

### RuleDialog — Match 조건 폼 설계

복잡한 match 조건을 접을 수 있는 "Advanced Conditions" 섹션으로 구분:

```
기본 정보:
  - Name (required)
  - Priority (number, default: 100)
  - Enabled (toggle)

조건 (Advanced — 기본 접힘):
  - Model Name (exact)
  - Model Prefix
  - Model Regex
  - Min Context Tokens / Max Context Tokens
  - Has Tools (checkbox)
  - Metadata (키-값 동적 추가/삭제)
  - Key ID / User ID / Team ID / Org ID (UUID 입력)

전략 & 타겟:
  - Strategy (select: direct/weighted/least_cost/failover/quality_first)
  - Targets (동적 배열: provider + model + weight?)
    - weighted 전략일 때만 weight 입력란 표시
    - direct 전략일 때 1개만 허용
```

### Strategy별 타겟 입력 규칙

| Strategy | 타겟 수 | weight 필드 |
|---|---|---|
| `direct` | 정확히 1개 | 없음 |
| `weighted` | 2개 이상 | 필수 (합산 100 검증) |
| `least_cost` | 2개 이상 | 없음 |
| `failover` | 2개 이상 | 없음 (순서가 우선순위) |
| `quality_first` | 2개 이상 | 없음 |

## 10. Risks

| 위험 | 확률 | 대응 |
|---|---|---|
| Match 조건 폼이 복잡하여 구현 비용 증가 | 중간 | "Advanced Conditions" 접힘 처리로 기본 UX 단순화; 복잡한 UUID 필드는 텍스트 입력으로 처리 |
| weighted strategy의 weight 합산 100 검증 로직 | 낮음 | Submit 전 클라이언트 검증으로 방어 |
| DryRun 결과의 `matched_rule` 필드가 ID인지 이름인지 불명확 | 중간 | 런타임 확인 후 ID면 목록에서 이름 조회, 이름이면 직접 표시 |
| `routingRules.delete` 가 `apiFetch`가 아닌 raw `fetch` 사용 | 낮음 | 401 시 자동 리다이렉트 없음 주의; 오류 발생 시 수동 에러 처리 필요 |
| 규칙 수가 많을 경우 클라이언트 정렬 성능 | 낮음 | 현재 백엔드가 정렬을 보장하지 않으면 클라이언트에서 `sort by priority`; 수백 건 이내에서는 문제없음 |

## 11. Edge Cases

- 규칙 목록이 비어 있을 때 → empty state ("No routing rules yet" + Add Rule 버튼)
- `direct` strategy에서 타겟을 2개 이상 추가하려 할 때 → 추가 버튼 비활성화
- `weighted` strategy에서 weight 합산이 100 미만/초과 시 → Submit 비활성 + 경고 문구
- match 조건이 모두 비어있는 규칙 ("catch-all") → 허용하되, Match 컬럼에 "Any" 표시
- DryRun 결과에서 `targets`가 빈 배열일 때 → "No targets available" 메시지
- UUID 형식이 아닌 값을 key_id/user_id 등에 입력했을 때 → 클라이언트 사전 검증
- 규칙 삭제 후 동일 priority의 다른 규칙이 있어도 재정렬은 서버 책임 (클라이언트는 목록 refetch만)
