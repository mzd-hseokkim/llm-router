## 1. Background

LLM Router Gateway는 OpenAI-compatible 엔드포인트를 제공한다. 현재 `/v1/chat/completions`, `/v1/embeddings`, `/v1/models`는 완전히 구현되어 있으나, `/v1/completions` (Legacy Text Completion API)는 핸들러 stub만 존재한다.

### 현재 상태 분석

| 컴포넌트 | 위치 | 상태 |
|---|---|---|
| 라우터 등록 | `internal/gateway/router/router.go:106` | ✅ 이미 등록됨 (`r.Post("/completions", comp.Handle)`) |
| 핸들러 파일 | `internal/gateway/handler/completions.go` | ⚠️ Stub (TODO 주석, 빈 응답 반환) |
| 기존 테스트 | `internal/gateway/handler/completions_test.go` | ⚠️ Stub 동작 기준으로만 작성 |
| Provider 인터페이스 | `internal/provider/interface.go` | ✅ Chat only (`ChatCompletion`, `ChatCompletionStream`) |
| SSE 스트리밍 | `internal/gateway/proxy/stream.go` | ✅ Chat chunk 포맷 (`chat.completion.chunk`) |

**핵심 발견**: 라우터 등록은 이미 완료되어 있다. 주요 작업은 핸들러의 실제 로직 구현이다.

## 3. Architecture Decision: Prompt-to-Chat Bridging

### 결정: Provider 인터페이스 확장 대신 Chat Bridging 사용

**배경**: OpenAI의 `/v1/completions`는 deprecated이며 현대 LLM은 chat 포맷이 표준이다. 대부분의 provider(Anthropic, Gemini, Cohere 등)는 텍스트 완성 전용 API가 없다.

**전략**: `CompletionRequest.prompt`를 `ChatCompletionRequest.messages[{role:"user", content: prompt}]`로 변환 후 기존 fallback router를 재사용한다.

```
CompletionRequest{prompt: "Once upon a time"}
    ↓ promptToMessages()
ChatCompletionRequest{messages: [{role:"user", content:"Once upon a time"}]}
    ↓ fallback.Router.Execute()
ChatCompletionResponse{choices:[{message:{content:"..."}}]}
    ↓ chatToCmplResponse()
CompletionResponse{choices:[{text:"..."}]}
```

**장점**:
- 코드 변경 최소화 (8개 provider adapter 수정 불필요)
- 기존 fallback, circuit breaker, telemetry 재사용
- 유지보수성 우수

**트레이드오프**:
- OpenAI의 실제 `/v1/completions` 엔드포인트 pass-through 불가 (chat endpoint로 변환됨)
- `logprobs`, `echo` 등 text-completion 전용 파라미터 미지원

## 4. Acceptance Criteria

```
Given: POST /v1/completions {"model": "openai/gpt-4o", "prompt": "Hello"}
When:  유효한 API 키로 요청
Then:  200 OK, CompletionResponse{object:"text_completion", choices:[{text: "..."}]}

Given: POST /v1/completions {"model": "openai/gpt-4o", "prompt": "Hello", "stream": true}
When:  유효한 API 키로 요청
Then:  200 OK, text/event-stream, SSE chunks with object:"text_completion"

Given: POST /v1/completions {"model": "openai/gpt-4o"}  (prompt 없음)
When:  요청
Then:  400 Bad Request

Given: POST /v1/completions {"model": "unknown/model", "prompt": "Hello"}
When:  요청
Then:  400 Bad Request, "Invalid model" 오류

Given: Provider가 오류 반환
When:  요청
Then:  적절한 HTTP 상태 코드와 에러 응답 (provider.GatewayError → HTTP status mapping)
```

## 7. Risks & Mitigations

| 리스크 | 가능성 | 영향 | 대응 |
|---|---|---|---|
| OpenAI native `/v1/completions` pass-through 불가 | 낮음 | 낮음 | Chat bridging 명시적으로 문서화 |
| prompt가 `[]string`일 때 첫 번째만 처리 | 중간 | 낮음 | 현재 요구사항 범위 내; 로그로 경고 |
| 스트리밍 SSE 포맷 불일치 | 중간 | 중간 | 단위 테스트로 청크 포맷 검증 |
| `fallback.Router.Execute` 시그니처가 `*ChatCompletionRequest` 기대 | 없음 | 없음 | bridge req를 명시적으로 생성 |

---

## 8. Edge Cases

1. **prompt가 `[]string`**: 첫 번째 문자열만 user message로 사용 (경고 로그)
2. **prompt가 공백 문자열**: validation에서 `len(trim) == 0` 체크 후 400
3. **max_tokens = 0**: 이미 validation에서 처리 (`<= 0` → 400)
4. **stream=true 이지만 client가 SSE 미지원**: `http.Flusher` 확인, 500 반환
5. **provider가 빈 content 반환**: `choices[0].text = ""`, `finish_reason = "stop"` 그대로 반환
