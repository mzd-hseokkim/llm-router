## 1. Background

LLM Router Admin UI에 A/B 테스트 관리 페이지(`/ab-tests`)가 없어 운영 불가 상태다. 백엔드 API(`/admin/ab-tests/*`)와 `lib/api.ts`의 `abTests` 객체는 이미 완성되어 있으며, UI만 없는 상황이다.

## 2. Backend API Contract

### 엔드포인트 목록

| Method | Endpoint | 설명 |
|---|---|---|
| `GET` | `/admin/ab-tests` | 실험 목록 → `{ data: Experiment[] }` |
| `POST` | `/admin/ab-tests` | 실험 생성 → `Experiment` (status: draft) |
| `GET` | `/admin/ab-tests/:id` | 단일 실험 조회 → `Experiment` |
| `GET` | `/admin/ab-tests/:id/results` | 통계 분석 결과 → `AnalysisResult` |
| `POST` | `/admin/ab-tests/:id/start` | draft/paused → running |
| `POST` | `/admin/ab-tests/:id/pause` | running → paused |
| `POST` | `/admin/ab-tests/:id/stop` | * → stopped |
| `POST` | `/admin/ab-tests/:id/promote` | winner 지정 → completed |

### 핵심 타입 (Backend `internal/abtest/experiment.go`)

```go
type Status string  // "draft" | "running" | "paused" | "completed" | "stopped"

type TrafficSplit struct {
    Variant string `json:"variant"`
    Model   string `json:"model"`
    Weight  int    `json:"weight"` // 0-100; 합산 100
}

type Target struct {
    TeamIDs    []string `json:"team_ids,omitempty"`
    SampleRate float64  `json:"sample_rate"` // 0.0-1.0
}

type Experiment struct {
    ID              string         `json:"id"`
    Name            string         `json:"name"`
    Status          Status         `json:"status"`
    TrafficSplit    []TrafficSplit  `json:"traffic_split"`
    Target          Target         `json:"target"`
    SuccessMetrics  []string       `json:"success_metrics"`
    MinSamples      int            `json:"min_samples"`
    ConfidenceLevel float64        `json:"confidence_level"`
    StartAt         *time.Time     `json:"start_at,omitempty"`
    EndAt           *time.Time     `json:"end_at,omitempty"`
    Winner          string         `json:"winner,omitempty"`
    CreatedAt       time.Time      `json:"created_at"`
    UpdatedAt       time.Time      `json:"updated_at"`
}
```

### 분석 결과 타입 (`internal/abtest/analyzer.go`)

```go
type AnalysisResult struct {
    TestID         string                         `json:"test_id"`
    Status         Status                         `json:"status"`
    Winner         string                         `json:"winner,omitempty"`
    Results        map[string]*VariantResult      `json:"results"`
    Significance   map[string]*MetricSignificance `json:"statistical_significance,omitempty"`
    Recommendation string                         `json:"recommendation,omitempty"`
}

type VariantResult struct {
    Model             string  `json:"model"`
    Samples           int     `json:"samples"`
    LatencyP95Ms      float64 `json:"latency_p95_ms"`
    AvgCostPerRequest float64 `json:"avg_cost_per_request"`
    ErrorRate         float64 `json:"error_rate"`
}

type MetricSignificance struct {
    PValue         float64 `json:"p_value"`
    Significant    bool    `json:"significant"`
    ImprovementPct float64 `json:"improvement_pct"`
}
```

### 기존 `lib/api.ts` 타입 (이미 완성 — 수정 불필요)

```typescript
export interface ABTestTrafficSplit {
  variant: string;
  weight: number;
}

export interface ABTestTarget {
  model: string;
  sample_rate?: number;
}

export interface ABTest {
  id: string;
  name: string;
  status: string;        // "draft" | "running" | "paused" | "completed" | "stopped"
  traffic_split: ABTestTrafficSplit[];
  target: ABTestTarget;
  success_metrics?: string[];
  min_samples?: number;
  confidence_level?: number;
  start_at?: string;
  end_at?: string;
}

export const abTests = {
  list: () => apiFetch<{ data: ABTest[] }>("/ab-tests").then((r) => r.data),
  get: (id: string) => apiFetch<ABTest>(`/ab-tests/${id}`),
  create: (payload: CreateABTestPayload) => apiFetch<ABTest>("/ab-tests", ...),
  results: (id: string) => apiFetch<Record<string, unknown>>(`/ab-tests/${id}/results`),
  start: (id: string) => apiFetch<{ status: string }>(`/ab-tests/${id}/start`, ...),
  pause: (id: string) => apiFetch<{ status: string }>(`/ab-tests/${id}/pause`, ...),
  stop: (id: string) => apiFetch<{ status: string }>(`/ab-tests/${id}/stop`, ...),
  promote: (id, winner) => apiFetch<{ status: string; winner: string }>(`/ab-tests/${id}/promote`, ...),
};
```

**주의**: `lib/api.ts`의 `ABTestTarget`에 `model` 필드가 있으나, 백엔드 `Target`에는 `model`이 없다 (모델은 `TrafficSplit.Model`에 있음). 실제 응답 JSON을 확인하고 타입 불일치 시 페이지 내부에서 처리해야 한다.

## 3. Scope

### In Scope

- `app/(admin)/ab-tests/page.tsx` — A/B 테스트 목록 + CRUD + 결과 뷰 구현
- `components/Sidebar.tsx` — "A/B Tests" 링크 추가 (`/ab-tests`)
- 실험 목록 (상태 배지: draft/running/paused/completed/stopped)
- 실험 생성 폼 (이름, traffic_split 2개 이상, target, min_samples, confidence_level)
- 상태 전환 버튼 (Start / Pause / Stop / Promote)
- 결과 뷰 — 인라인 패널: variant별 응답시간(P95), 비용, 에러율 비교 테이블
- 통계적 유의성 표시 (`statistical_significance`)
- 승자 전환 버튼 (Promote) — variant 선택 후 POST `/promote`
- 컴포넌트 단위 테스트 (`app/(admin)/__tests__/ABTestsPage.test.tsx`)

### Out of Scope

- `lib/api.ts` 수정 — 이미 완성 (타입 불일치 발생 시 페이지 내 로컬 타입으로 처리)
- 백엔드 수정
- 차트 라이브러리 도입 (번들 크기 최소화, 테이블 기반 비교로 구현)
- 실험 수정(PUT) — 백엔드 PUT 엔드포인트가 Jira 명세에 없음; Create + 상태 전환으로 충분
- 삭제(DELETE) — concluded/stopped 상태 보존이 중요하므로 제외

## 4. Acceptance Criteria

### AC-1: 실험 목록
- **Given** Admin UI의 `/ab-tests` 페이지에 접속했을 때
- **When** 페이지가 로드되면
- **Then** 실험 목록(이름, 상태, traffic_split 요약, 생성일)이 테이블로 표시된다
- **And** 상태에 따른 색상 배지(draft=gray, running=green, paused=yellow, completed=blue, stopped=red)가 표시된다

### AC-2: 실험 생성
- **Given** "+ New A/B Test" 버튼을 클릭했을 때
- **When** 이름과 최소 2개의 traffic_split(variant+model+weight)을 입력하고 저장하면
- **Then** 새 실험이 draft 상태로 목록에 추가되고 모달이 닫힌다

### AC-3: 상태 전환 버튼
- **Given** 실험 목록에서 특정 실험의 액션 버튼을 클릭했을 때
- **When** Start(draft/paused) / Pause(running) / Stop(running/paused)를 누르면
- **Then** 해당 상태로 전환되고 목록이 갱신된다

### AC-4: 결과 뷰
- **Given** 실험 행의 "Results" 버튼을 클릭했을 때
- **When** 결과 패널이 열리면
- **Then** 각 variant의 samples, latency_p95_ms, avg_cost_per_request, error_rate가 테이블로 표시된다
- **And** statistical_significance가 있으면 metric별 p-value, improvement_pct가 표시된다
- **And** recommendation 문자열이 있으면 표시된다

### AC-5: Promote (승자 전환)
- **Given** 실험이 running 또는 completed 상태일 때 "Promote" 버튼을 클릭했을 때
- **When** variant를 선택하고 확인하면
- **Then** 해당 variant가 winner로 설정되고 실험이 completed 상태로 전환된다

### AC-6: Sidebar 링크
- **Given** Admin UI 좌측 Sidebar가 표시된 상태에서
- **When** "A/B Tests" 링크를 클릭하면
- **Then** `/ab-tests` 페이지로 이동하고 링크가 active 스타일로 표시된다

## 10. Risks

| 위험 | 확률 | 대응 |
|---|---|---|
| `lib/api.ts`의 `ABTestTarget.model` vs 백엔드 `TrafficSplit.Model` 타입 불일치 | 중간 | 페이지 내 로컬 타입 정의로 `AnalysisResult` 처리; `ABTest` 타입은 그대로 사용 |
| ATL-290과 `Sidebar.tsx` 충돌 | 중간 | ATL-290 merge 전 rebase 또는 합산 커밋으로 조율 |
| `/results` API가 샘플 수 부족 시 `statistical_significance` 없이 응답 | 낮음 | `significance` 필드가 없을 때 graceful empty state 처리 |
| weight 합산 100 검증 실패 시 백엔드 422 | 낮음 | 프론트엔드에서 사전 검증으로 방어 |
| 실험이 많을 경우 목록 성능 | 낮음 | 현재 백엔드가 페이지네이션 미지원(단순 List); 추후 ATL-469 패턴 적용 가능 |

---

## 11. Edge Cases

- 실험 목록이 비어 있을 때 → empty state ("No A/B tests yet" + 생성 버튼)
- traffic_split weight 합산이 100이 아닐 때 → Submit 버튼 disabled + 경고 문구
- results API 응답에 samples=0인 variant → "0 samples (no data yet)" 표시
- `recommendation`이 빈 문자열이거나 없을 때 → 섹션 미표시
- 이미 completed/stopped 상태의 실험 → 상태 전환 버튼 없음, Results만 표시
- Promote 시 winner variant가 이미 설정된 경우 → 현재 winner 강조 표시 후 재선택 가능

## 12. Open Questions

1. **ATL-290 조율**: Sidebar.tsx 수정이 ATL-290과 겹치는데, 어느 쪽을 먼저 merge할지 사전 조율이 필요하다.
2. **`lib/api.ts`의 `ABTestTarget.model` 필드**: 백엔드 `Target` 구조에는 `model`이 없고 모델은 `TrafficSplit.Model`에 있다. `abTests.results()`의 실제 응답 JSON을 런타임에서 확인하여 페이지 내 타입 정의를 확정해야 한다.
3. **페이지네이션**: 현재 백엔드 List API는 pagination이 없다. 실험이 많아질 경우 ATL-469 패턴으로 추가할 수 있으나, 이번 태스크 범위 외다.
