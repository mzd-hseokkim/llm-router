# ATL-289 기획 문서: Admin UI Dashboard 보완

## 1. 배경 및 목적

### 현황
Admin UI의 Dashboard(`admin-ui/app/(admin)/page.tsx`)에는 현재 다음 요소만 구현되어 있다:
- 4개 통계 카드: Monthly Cost, Total Requests, Active Keys, Active Provider Keys
- 2개 차트: Top Spenders Bar Chart, Provider Key Status Pie Chart

### 목적
운영자가 Gateway의 상태를 Dashboard 한 화면에서 빠르게 파악할 수 있도록 3가지 보완 지표를 추가한다:
1. **Circuit Breaker 상태 요약 위젯** — Provider별 OPEN/CLOSED 현황
2. **활성 예산 초과 임박 경고 배너** — 임계값 초과 Provider Key 경고
3. **오늘의 캐시 히트율 지표** — 당일 exact cache hit/miss 비율

---

## 2. 범위 (Scope)

### In Scope
- `admin-ui/app/(admin)/page.tsx`: 3개 UI 위젯 추가
- `internal/gateway/handler/admin_usage.go` (또는 `admin_cache.go`): 캐시 히트율 조회 엔드포인트 추가
- `admin-ui/lib/api.ts`: 캐시 통계 API 클라이언트 함수 추가

### Out of Scope
- 신규 DB 마이그레이션 (기존 `request_logs.cache_hit` 컬럼 활용)
- Circuit Breaker 또는 Budget 관련 새 백엔드 엔드포인트 (기존 API 사용)
- 시맨틱 캐시 히트율 (현재 `internal/cache/exact`만 대상)
- 알림 발송 로직 변경 (기존 alerting 시스템 유지)
- 모바일 레이아웃 최적화

## 3. Acceptance Criteria

### AC-1: Circuit Breaker 위젯
- Given: `/admin/circuit-breakers` API 응답에 Provider 목록이 있을 때
- When: Dashboard 페이지를 열면
- Then: 각 Provider의 Circuit Breaker 상태(CLOSED/OPEN/HALF_OPEN)가 색상 배지로 표시된다
- Then: 데이터가 없으면 "No circuit breakers tracked" 메시지가 표시된다
- Then: 30초마다 자동 갱신된다

### AC-2: 예산 초과 임박 경고 배너
- Given: Provider Key 중 `current_month_spend / monthly_budget_usd >= threshold`인 항목이 있을 때
- When: Dashboard 페이지를 열면
- Then: 해당 키 이름과 사용률(%)이 표시되는 경고 배너가 렌더링된다
- Then: `monthly_budget_usd`가 null/0인 키는 배너에서 제외된다
- Then: 임계값은 `alerts.getConfig()`의 `budget_threshold_pct`를 사용하며, 조회 실패 시 80%를 기본값으로 사용한다
- Then: 경고 항목이 없으면 배너가 표시되지 않는다

### AC-3: 오늘의 캐시 히트율 지표
- Given: `GET /admin/cache/stats` 엔드포인트가 추가되었을 때
- When: Dashboard 페이지를 열면
- Then: "Today's Cache Hit Rate: XX.X%" 형태의 StatCard가 표시된다
- Then: 오늘 캐시 조회 건수가 0이면 "N/A"를 표시한다
- Then: 30초마다 자동 갱신된다

## 4. 기술 분석

### 4-1. Circuit Breaker 위젯 (프론트엔드 전용)
- **API**: `GET /admin/circuit-breakers` → 이미 구현됨
- **프론트엔드 API 클라이언트**: `circuitBreakers.list()` → `lib/api.ts`에 이미 존재
- **응답 구조**: `{ provider, state, failure_count, last_failure, reset_time }`
- **state 값**: `"closed"`, `"open"`, `"half_open"` (circuit_breaker.go 참조)
- **추가 작업**: `page.tsx`에 위젯 컴포넌트만 추가

### 4-2. 예산 경고 배너 (프론트엔드 전용)
- **API**: `providerKeys.list()` → 각 ProviderKey에 `monthly_budget_usd`와 `current_month_spend` 포함
- **임계값**: `alerts.getConfig()` → `conditions.budget_threshold_pct` (0–100 정수)
- **계산**: `current_month_spend / monthly_budget_usd * 100 >= threshold`
- **추가 작업**: `page.tsx`에 배너 컴포넌트 추가

### 4-3. 캐시 히트율 지표 (백엔드 + 프론트엔드)

#### 백엔드: `GET /admin/cache/stats`
- `request_logs` 테이블에는 `cache_hit BOOLEAN` 컬럼이 존재 (migration 004)
- 오늘(`NOW() AT TIME ZONE 'UTC'` 기준 날짜 시작 ~ 현재) 조건으로 집계
- 쿼리:
  ```sql
  SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE cache_hit = true) AS hits
  FROM request_logs
  WHERE timestamp >= date_trunc('day', NOW() AT TIME ZONE 'UTC')
  ```
- 응답: `{ total: int, hits: int, hit_rate: float64 }`
- 구현 위치: `internal/gateway/handler/admin_cache.go`에 `Stats` 메서드 추가
- 라우터 등록: `cmd/gateway/main.go` (또는 라우터 설정 파일) 확인 후 추가

#### 프론트엔드
- `lib/api.ts`에 `cacheStats` 함수 추가:
  ```ts
  cacheStats: () => apiFetch<{ total: number; hits: number; hit_rate: number }>("/cache/stats")
  ```
- `page.tsx`에 `StatCard` 추가

## 6. 의존성 및 위험

### 의존성
- T3는 T1 완료 후 API 경로 확정 필요
- T4, T5, T6는 T3 완료 후 병렬 구현 가능

### 위험 요소
| 위험 | 영향 | 대응 |
|------|------|------|
| `request_logs` 파티션 날짜 범위 오류 | 캐시 히트율 쿼리 실패 | 에러 시 `hit_rate: null` 반환, UI에서 "N/A" 표시 |
| Circuit Breaker가 아무것도 추적 안 함 | 빈 배열 반환 | "No circuit breakers tracked" 빈 상태 UI |
| `alerts.getConfig()` 실패 | 임계값 조회 불가 | fallback으로 80% 하드코딩 |
| DB 연결 없이 cache stats 조회 | 500 에러 | admin_cache.go가 DB 접근 방법 없는 경우 → admin_usage.go에 메서드 추가하는 방향으로 전환 |

## 7. Edge Cases

- **Circuit Breaker state 값이 예상 외**: 알 수 없는 state는 회색 배지(Unknown)로 표시
- **budget_usd = 0 또는 null**: 배너 계산에서 제외 (divide by zero 방지)
- **today total = 0**: 캐시 히트율 "N/A" 표시
- **API 오류 시**: 각 위젯은 독립적으로 에러 상태를 처리하며, 하나가 실패해도 다른 위젯에 영향 없음
