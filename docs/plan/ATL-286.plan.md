# ATL-286: Admin UI: 알림 설정 페이지 (/alerts)

**작성일**: 2026-02-23
**우선순위**: 주요 (Major)
**브랜치**: `feature/ATL-286`

---

## Background

백엔드 알림 API(`GET/PUT /admin/alerts/config`, `POST /admin/alerts/test`, `GET /admin/alerts/history`)는 이미 완성되어 있다. `lib/api.ts`에도 `alerts.test()`와 `alerts.history()`는 존재하지만, 설정 조회/수정(`GET/PUT /admin/alerts/config`)이 없다. Admin UI에 `/alerts` 페이지가 없어서 Slack/Email/Webhook 채널 설정, 알림 조건 편집, 테스트 발송, 히스토리 확인이 불가능한 상태다.

- 알림 수신 대상(사용자) 관리

## Technical Design

### 1. `lib/api.ts` 변경

**추가할 타입:**

```typescript
export interface AlertChannelConfig {
  slack?: { webhook_url: string; enabled: boolean };
  email?: { addresses: string[]; enabled: boolean };
  webhook?: { url: string; enabled: boolean };
}

export interface AlertConditions {
  budget_threshold_pct?: number;   // 예산 소진 임박 % (예: 80)
  error_rate_threshold?: number;   // 에러율 임계값 % (예: 5)
  latency_threshold_ms?: number;   // 응답시간 임계값 ms (예: 2000)
}

export interface AlertConfig {
  channels: AlertChannelConfig;
  conditions: AlertConditions;
  enabled: boolean;
}
```

**`alerts` 객체에 추가할 메서드:**

```typescript
export const alerts = {
  // 기존 유지
  test: ...,
  history: ...,

  // 추가
  getConfig: () => apiFetch<AlertConfig>("/alerts/config"),
  updateConfig: (payload: Partial<AlertConfig>) =>
    apiFetch<AlertConfig>("/alerts/config", {
      method: "PUT",
      body: JSON.stringify(payload),
    }),
};
```

### 2. `app/(admin)/alerts/page.tsx` 구조

guardrails 페이지의 패턴을 따라 SectionCard 기반 레이아웃 사용:

```
/alerts
├── 페이지 헤더 ("알림 설정")
├── SectionCard: "채널 설정"
│   ├── Slack (toggle 활성화 + webhook URL 입력 + 테스트 버튼)
│   ├── Email (toggle 활성화 + 수신 주소 입력 + 테스트 버튼)
│   └── Webhook (toggle 활성화 + URL 입력 + 테스트 버튼)
├── SectionCard: "알림 조건"
│   ├── 예산 임박 알림 (%) — number input
│   ├── 에러율 임계값 (%) — number input
│   └── 응답시간 임계값 (ms) — number input
├── [저장] 버튼 (PUT /alerts/config)
└── SectionCard: "최근 발송 히스토리"
    └── 테이블: 발송 시각 / 이벤트 타입 / 채널 / 상태 / 오류
```

**데이터 흐름:**

```typescript
// 설정 로드
const { data: config } = useQuery({
  queryKey: ["alerts-config"],
  queryFn: () => alerts.getConfig(),
});

// 설정 저장
const saveMutation = useMutation({
  mutationFn: (payload: Partial<AlertConfig>) => alerts.updateConfig(payload),
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ["alerts-config"] }),
  onError: (e: Error) => setError(e.message),
});

// 테스트 발송
const testMutation = useMutation({
  mutationFn: (channel: string) => alerts.test(channel),
  onSuccess: (data) => setTestResult({ ok: true, message: data.status }),
  onError: (e: Error) => setTestResult({ ok: false, message: e.message }),
});

// 히스토리
const { data: history = [] } = useQuery({
  queryKey: ["alerts-history"],
  queryFn: () => alerts.history(50),
  refetchInterval: 30_000,
});
```

## Risks & Edge Cases

| 위험 | 대응 |
|------|------|
| 백엔드가 `AlertConfig`의 정확한 필드명을 달리 쓸 수 있음 | 첫 `getConfig()` 호출 응답 보고 타입 조정 |
| Email 수신 주소가 복수(배열)일 때 UI 입력 방식 | textarea + 줄바꿈 파싱 또는 콤마 구분 입력으로 처리 |
| 테스트 발송 중 네트워크 오류 | `testMutation.isError` 상태로 인라인 에러 메시지 표시 |
| `getConfig()` 응답 없을 때(204 또는 빈 객체) 폼 초기화 | 빈 객체 기본값 + optional chaining으로 처리 |
